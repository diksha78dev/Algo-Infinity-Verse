💾 Why I/O, Not Comparisons, Is the Real Cost

📄

Phase 1 — Run Generation

Read memory-budget-sized chunks from disk, sort each in RAM (any in-memory algorithm works), write the sorted chunk back as a "run." A file of N elements with memory budget M produces ⌈N/M⌉ initial sorted runs.

🔀

Phase 2 — K-Way Merge

Keep one small buffer per run being merged, always output the smallest buffered value, then refill that run's buffer from disk. With memory for M-1 run buffers plus 1 output buffer, you can merge M-1 runs per pass.

💿

Why I/O Dominates

Disk reads/writes are orders of magnitude slower than RAM comparisons. External sort algorithms are designed to minimize the number of PASSES over the data — each pass reads and writes every element once — not to minimize comparisons.

🏢

Real Databases

PostgreSQL's `work_mem` setting directly controls this memory budget — too small, and a large `ORDER BY` triggers many merge passes and becomes disk-bound. This is why increasing `work_mem` is a classic query-performance fix for sort-heavy queries.
