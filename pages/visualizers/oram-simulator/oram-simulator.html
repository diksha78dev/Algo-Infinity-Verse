Oblivious RAM Simulator

Encryption hides what you store. ORAM hides what you access.

An adversary watching your memory doesn't need to read the data — access patterns alone reveal which operation you performed. Path ORAM fixes this by fetching an entire root-to-leaf path on every single access, making all access patterns indistinguishable. Every access looks identical to the observer. This is how Signal, Intel SGX enclaves, and privacy-preserving databases prevent pattern-based leakage.

🔐 Why ORAM Matters

👁️

The Pattern Leakage Problem

Imagine your health records are encrypted in a cloud database. An adversary can't read them — but if they watch which records you access after visiting a doctor, they can infer your diagnosis. The ACCESS PATTERN leaks information even when the data is perfectly encrypted. ORAM closes this gap.

🌲

Why a Tree Structure?

Path ORAM stores blocks in a binary tree where every block is assigned to a random leaf. Fetching the root-to-leaf path always reads O(log N) buckets — a fixed, predictable cost. The block can be anywhere on its assigned path, and after every access, its path assignment is re-randomized, making future accesses to the same block look completely different.

🔒

Intel SGX Uses This

Intel SGX enclaves run code in a hardware-protected region, but memory accesses outside the enclave are still visible. Oblivious RAM is used inside SGX applications to prevent the untrusted OS from learning which data the enclave is processing, even though it can watch every memory bus transaction.

⚖️

The Cost of Obliviousness

Path ORAM has O(log N) overhead per access — a 4-level tree touches 4 buckets, each holding Z=4 blocks. For N=2^20 blocks, each access touches 20 buckets instead of 1. This is a 20× bandwidth overhead. In practice, recursive ORAM and optimized variants like RingORAM reduce this to near-constant overhead.
